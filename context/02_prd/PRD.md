## 1. 项目背景

多店铺积分商城系统是一个**多租户电商平台**，以**积分作为唯一货币**的商品兑换系统。每个店铺拥有独立的数据和运营管理能力，平台方（管理员）可俯瞰所有店铺运营数据。

系统采用**表级别租户隔离**架构（通过 shop_id 区分），三层用户角色体系：
- **平台层**：管理员（平台方），全局数据，可查看所有店铺数据
- **店铺层**：店铺用户（每个店铺的管理员），本店数据完全独立
- **用户层**：普通用户（店铺的消费者），只能访问所属店铺的数据

核心业务能力包括：账号注册与审核、店铺入驻与审核、商品管理与分类、积分兑换订单、用户地址簿管理、站内消息通知等。

## 2. 项目目标

### 核心目标

- 构建支持多店铺独立运营的积分兑换平台，实现店铺级数据隔离
- 提供完整的商品管理、用户管理、订单处理能力，支持实物/虚拟商品分类
- 建立以积分为唯一货币的兑换体系，积分永不过期，店铺用户手动维护
- 实现站内消息通知机制，保障审核结果通知、订单超时提醒

### 管控诉求

- 管理员可全局监控所有店铺运营数据（店铺数、用户数、订单数、积分发放）
- 支持账号审核、店铺审核、店铺冻结/解冻等管控能力
- 普通用户最多5个收货地址，支持默认地址设置，实物兑换自动带入

### 约束条件

- 积分是唯一货币，无现金支付
- 邀请码永久有效，重新生成后旧码立即失效
- 普通用户最多5个收货地址，仅1个默认地址
- 店铺间数据完全隔离，不可跨店查询

## 4. 功能模块详细说明

### 4.1 平台管理

#### 4.1.1 模块业务目标

为平台管理员提供全局运营管控能力，实现店铺接入审核、跨店铺用户管理、全局数据监控。管理员不直接管理积分，由店铺用户维护。

#### 4.1.2 模块业务流程

**账号注册流程**：店铺用户提交注册申请 → 管理员审核 → 通过（开通登录权限）/ 拒绝（可重新申请）

**店铺申请流程**：店铺用户登录后填写申请信息 → 提交申请 → 管理员审核 → 通过（可上架商品）/ 拒绝（可重新申请）/ 冻结（需重新申请）

**密码重置流程**：管理员选择用户 → 确认重置 → 系统生成随机密码 → 弹框显示 → 线下告知用户

#### 4.1.3 功能列表

| 功能 | 说明 |
|:--|:--|
| 全局数据统计 | 查看所有店铺数量、用户总数、订单总量、积分发放统计（实时） |
| 店铺审核 | 查看店铺申请列表，审核通过/拒绝 |
| 店铺管理 | 冻结/解冻店铺，查看店铺列表及详情 |
| 用户账号管理 | 跨店铺查看店铺用户和普通用户列表，强制重置任何用户密码 |
| 个人信息管理 | 修改个人信息、修改密码 |
| 邀请码管理 | 生成邀请码用于邀请店铺用户（需审核账号） |
| 站内消息 | 查看系统发送的消息通知 |

#### 4.1.4 操作人员定义

**平台管理员**：平台超级管理员，系统初始化时内置第一个账号
- 负责系统全局数据监控、店铺审核、跨店铺用户账号管理
- 可生成邀请码邀请店铺用户
- 可强制重置任何用户密码
- 不直接管理积分和店铺商品订单

#### 4.1.5 对接的外部系统

| 外部系统 | 对接目的 |
|:--|:--|
| 文件存储服务 | 存储店铺Logo等图片文件 |

#### 4.1.6 模块边界和依赖关系

本模块为平台管理层依赖文件存储服务等外部能力。被店铺运营模块依赖：管理员审核通过后店铺才可正常运营。

---

### 4.2 店铺运营

#### 4.2.1 模块业务目标

为店铺用户提供完整的本店商品管理、用户管理、订单处理、积分维护能力。店铺用户负责本店具体运营，包括商品上架、订单处理、用户积分维护。

#### 4.2.2 模块业务流程

**商品管理流程**：创建商品（名称/图片/积分/库存/分类） → 设置单用户兑换限制 → 保存（下架状态） → 上架商品

**普通用户创建流程**：生成邀请码分发给消费者 → 普通用户注册 → 店铺用户审核 → 通过/拒绝；或直接创建用户 → 系统生成密码 → 线下分发

**订单处理流程**：普通用户下单 → 订单状态：已下单 → 店铺用户确认 → 已确认 → 填写快递信息发货 → 已发货 → 确认完成 → 已完成

**积分维护流程**：增加/扣除积分 → 填写操作理由 → 记录积分变动 → 更新用户余额

**地址簿管理流程**（普通用户）：进入地址簿 → 新增/编辑/删除/设默认地址 → 最多5个地址，设默认后下单自动带入

#### 4.2.3 功能列表

| 功能 | 说明 |
|:--|:--|
| 本店数据统计 | 查看店铺用户数、普通用户数、商品数、订单数、积分余额汇总 |
| 商品分类管理 | 创建/编辑/删除/查看本店商品分类（系统内置分类不可编辑） |
| 商品管理 | 创建/编辑/上架/下架/删除商品，设置积分价格、库存、单用户兑换限制 |
| 普通用户管理 | 查看用户列表及详情、生成/重新生成邀请码、创建/批量创建用户、审核用户注册 |
| 积分维护 | 增加/扣除用户积分（必填操作理由）、查看积分变动记录 |
| 密码重置 | 重置本店普通用户密码 |
| 订单处理 | 查看订单列表/详情、确认订单、发货（填写快递类型+单号）、完成订单、申请关闭订单 |
| 个人信息管理 | 修改个人信息、修改密码 |
| 邀请码管理 | 生成/重新生成邀请码（永久有效，重新生成后旧码失效） |
| 店铺管理 | 申请店铺、查看店铺状态、被拒绝/冻结后重新申请 |
| 站内消息 | 查看系统发送的消息通知 |

#### 4.2.4 操作人员定义

**店铺用户**：每个店铺的管理员账号（账号与店铺独立）
- 店铺「已通过」审核后才能上架商品
- 店铺「申请中」或「已冻结」时功能受限
- 只能操作本店数据

#### 4.2.5 对接的外部系统

| 外部系统 | 对接目的 |
|:--|:--|
| 文件存储服务 | 存储商品图片、店铺Logo |

#### 4.2.6 模块边界和依赖关系

本模块依赖系统内置商品分类（虚拟/实物）作为基础分类能力。商品图片通过文件存储服务管理。

---

### 4.3 用户兑换

#### 4.3.1 模块业务目标

为普通用户提供商品浏览、积分兑换下单、订单管理、地址簿管理能力。普通用户通过积分兑换商品，支持实物/虚拟商品分类，下单时自动带入默认收货地址。

#### 4.3.2 模块业务流程

**商品兑换流程**：浏览商品 → 选择商品 → 检查库存、兑换次数 → 确认兑换 → 实物商品填写收货信息（自动带入默认地址/从地址簿选择/手动输入） → 提交订单 → 积分扣除 → 库存扣减 → 订单状态：已下单

**订单关闭流程**：已下单订单 → 普通用户申请关闭 → 填写关闭理由 → 积分返还 → 订单状态：已关闭

**地址簿选择流程**：选择实物商品 → 点击兑换 → 弹出表单（自动带入默认地址） → 可点击「选择地址簿」→ 二次弹框选择地址 → 自动回显 → 可手动修改 → 确认兑换

#### 4.3.3 功能列表

| 功能 | 说明 |
|:--|:--|
| 商品浏览 | 查看商品列表（卡片/表格切换）、查看商品详情 |
| 商品兑换 | 使用积分兑换商品，实物商品填写收货信息 |
| 收货信息管理 | 自动带入默认地址、从地址簿选择、手动填写 |
| 订单管理 | 查看订单列表/详情、申请关闭已下单订单 |
| 个人信息管理 | 修改个人信息、查看积分余额、查看积分兑换历史、查看积分明细 |
| 地址簿管理 | 查看/新增/编辑/删除/设默认收货地址（最多5个） |
| 站内消息 | 查看系统发送的消息通知 |

#### 4.3.4 操作人员定义

**普通用户**：店铺的消费者，只能访问所属店铺数据
- 积分由店铺用户手动维护，不可自行充值
- 只能查看和操作自己的订单
- 最多5个收货地址，仅1个默认地址

#### 4.3.5 对接的外部系统

无直接外部系统依赖。

#### 4.3.6 模块边界和依赖关系

本模块为用户层，依赖店铺运营模块发布的商品数据。地址簿数据仅限普通用户个人管理。

### 4.4 系统支撑

#### 4.4.1 模块业务目标

为系统提供站内消息通知、文件存储、数据隔离等基础支撑能力，保障业务功能正常运行。

#### 4.4.2 模块业务流程

**站内消息流程**：系统触发通知条件 → 生成消息记录 → 用户查看我的消息

**订单超时提醒**：订单「已下单」超过7天未确认 → 站内消息提醒店铺用户

**文件上传流程**：用户上传文件 → 文件存储服务 → 返回文件URL → 关联业务表存储file_id

**数据隔离流程**：用户请求 → 系统通过shop_id过滤 → 返回本店数据

#### 4.4.3 功能列表

| 功能 | 说明 |
|:--|:--|
| 订单超时提醒 | 已下单超过7天未确认，系统发送站内消息提醒店铺用户 |
| 审核结果通知 | 账号/店铺/用户审核完成后发送站内消息通知 |
| 密码重置通知 | 密码被重置后发送站内消息通知用户 |
| 消息查看 | 用户查看系统发送给自己的所有站内消息 |
| 图片上传 | 商品主图、详情图、店铺Logo等文件存储 |
| 数据过滤 | 通过shop_id确保店铺间数据完全隔离 |

#### 4.4.4 操作人员定义

系统自动执行，无需人工操作。

#### 4.4.5 对接的外部系统

| 外部系统 | 对接目的 |
|:--|:--|
| 文件存储服务 | 存储商品图片、店铺Logo等静态文件 |

#### 4.4.6 模块边界和依赖关系

本模块为基础设施层，被所有业务模块依赖。文件存储服务为商品图片上传必需。

## 5. Feature拆分推荐

按推荐开发顺序排列，作为后续拆分的参考。建议按业务闭环拆分，每个Feature对应一个可独立交付的用户价值单元。

| 序号 | Feature名称 | 包含功能 | Feature类型 | 依赖Feature | 核心交付 |
|:--|:--|:--|:--|:--|:--|
| 1 | 系统基础设施 | 数据隔离框架、文件存储基础能力 | 基础设施 | - | 租户隔离、文件上传 |
| 2 | 账号与认证体系 | 登录、注册（含邀请码）、账号审核、密码重置 | 基础设施 | Feature1 | 完整账号链路、审核流程 |
| 3 | 站内消息通知 | 消息发送、消息查看、订单超时提醒（站内） | 通用服务 | Feature1 | 消息中心 |
| 4 | 平台管理层 | 管理员仪表盘、店铺管理（审核/冻结/解冻/列表）、跨店铺用户管理 | 业务功能 | Feature1, Feature2, Feature3 | 平台管控能力 |
| 5 | 店铺核心运营 | 商品分类管理（系统内置+商家自定义）、商品管理（CRUD/上下架） | 业务功能 | Feature1, Feature3 | 商品上架能力 |
| 6 | 店铺用户管理 | 普通用户管理（邀请码/创建/批量创建/审核注册）、积分维护（增加/扣除/记录） | 业务功能 | Feature2, Feature4, Feature5 | 用户运营能力 |
| 7 | 订单流程 | 订单处理（查看/确认/发货/完成/关闭）、订单超时提醒机制 | 业务功能 | Feature4, Feature5, Feature6 | 订单交付能力 |
| 8 | 用户端浏览与兑换 | 商品列表/详情（卡片/表格切换）、实物/虚拟兑换流程、收货信息（含地址簿选择与管理） | 业务功能 | Feature4, Feature6 | 用户兑换体验 |
| 9 | 用户端订单与个人中心 | 我的订单（查看/详情/关闭）、个人信息/积分查询 | 业务功能 | Feature8 | 用户自主管理 |

**拆分说明**：

1. **Feature2 专注账号认证**：登录、注册、邀请码、审核、密码重置，不包含消息通知。这一整个链路不能拆开，因为：
   - 邀请码生成后需要通过审核流程才能激活账号
   - 审核结果通过站内消息通知（调用 Feature3）
   - 密码重置后也通过站内消息通知（调用 Feature3）
   - 这些都是账号生命周期的组成部分

2. **Feature3 提供通用消息能力**：站内消息通知作为独立的基础设施层服务，被所有业务模块调用，包括账号审核通知、订单超时提醒等

3. **Feature8 包含地址簿管理**：地址簿是实物兑换的核心前置，必须在兑换前可用，因此放在用户端浏览与兑换中统一管理

4. **按业务闭环拆分**：商品→用户→订单形成三个独立闭环，每个可独立交付

## 6. 术语表

| 术语 | 定义 |
|:--|:--|
| 多租户 | 一种软件架构模式，多个租户共享同一个系统实例，但数据完全隔离。在本系统中，每个店铺为一个租户，拥有独立的数据。 |
| 积分商城 | 以积分作为唯一货币的商品兑换系统，用户通过积累的积分来兑换商品，不支持现金支付。 |
| 店铺用户 | 每个店铺的管理员账号，负责本店的商品管理、订单处理、用户积分维护等运营工作。账号与店铺是独立概念。 |
| 普通用户 | 店铺的消费者，通过积分兑换商品的用户群体。每个普通用户只属于一个店铺。 |
| 商家自定义分类 | 店铺用户自行创建的商品分类，用于对本店商品进行分组管理。 |
| 系统内置分类 | 系统预置的商品分类（如：虚拟、实物等），店铺用户不可编辑，仅可选择。 |
| 逻辑删除 | 一种数据删除方式，不是真正从数据库中删除数据，而是通过修改状态字段使数据不可见。 |
| 表级别隔离 | 多租户数据隔离的一种实现方式，所有租户的数据存在同一张表中，通过 shop_id 字段区分数据归属。 |
| 单用户最大兑换个数 | 商品的可重复兑换限制设置。默认0表示无限制，非0表示同一用户对该商品的最高兑换次数。 |
| 积分操作记录 | 店铺用户增加或扣除普通用户积分时，必须填写的操作原因说明，用于积分变动的可追溯。 |
| 订单超时提醒 | 订单状态为「已下单」超过7天未确认时，系统发送站内消息提醒店铺用户处理。 |
| 店铺冻结 | 管理员对店铺执行的强制封停操作。店铺冻结后，店铺用户无法操作商品和用户，商品自动下架。 |
| 邀请码 | 用于替代直接注册/创建流程的安全凭证。唯一对应一个目标账号类型（店铺用户/普通用户），生成后永久有效，重新生成后旧码立即失效。 |
| 强制重置密码 | 管理员或店铺用户对用户账号执行的密码重置操作。系统随机生成新密码，弹框显示后由执行者线下告知用户。 |
| 地址簿 | 普通用户管理的收货地址集合，用于实物商品下单时快速填写收货信息。最多5个地址 |
| 默认地址 | 普通用户设置的优先使用的收货地址，下单时自动带入。仅能设置1个 |
| 实物商品 | 需要物流配送的实体商品，下单时需填写收货人、手机、地址 |
| 虚拟商品 | 电子凭证类商品（如会员卡、点卡），无需物流，下单后直接发放 |
